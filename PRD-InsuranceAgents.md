# מסמך דרישות מוצר (PRD)
## InsuranceAgent — מערכת סוכני AI לשירות לקוחות בענף הביטוח

---

*גרסה 1.0 | תאריך: 28 באפריל 2026 | סטטוס: Draft*

---

## 1. תקציר מנהלים

InsuranceAgent היא מערכת סוכני AI רב-שכבתית המחליפה את שירות הלקוחות האנושי ב-Tier-1 וב-Tier-2 של חברת הביטוח. הלקוח מנהל שיחה טבעית עם Orchestrator Agent המזהה אותו, טוען את תיק הלקוח המלא, ומנתב לסוכן מומחה — מכירות, שימור, כספים, או תביעות. שכבת Supervisor Agent עצמאית פועלת ברקע, בוחנת כל תשובה לפני שליחתה, ומוודאת שהמערכת לא חורגת מגבולות ההלכה, הרגולציה, והאמת.

**מטרה עסקית:** חיסכון של 60% בעלות לאינטראקציה תוך שמירה על CSAT > 4.2/5, עמידה מלאה ברגולציית רשות שוק ההון, והגנה על הכנסות החברה דרך שימור לקוחות ומכירה חכמה.

## 3. רקע — הבעיה שאנחנו פותרים

### מצב נוכחי
מחלקת שירות הלקוחות של חברת הביטוח מנהלת עשרות אלפי אינטראקציות בחודש בערוצי טלפון, צ'אט ומייל. מרבית הפניות (כ-70%) הן שאלות שגרתיות: מצב תביעה, שינוי תשלום, פרטי פוליסה, חידוש. כל אלה אינן דורשות שיקול דעת אנושי — הן דורשות גישה למידע ועקביות.

**הכאב המרכזי:**
- עלות נציג אנושי בישראל: 25,000–35,000 ₪/חודש (כולל מעסיק)
- זמן המתנה ממוצע בשיחה: 8–12 דקות
- עקביות תשובות: נמוכה — כל נציג מפרש אחרת
- זמינות: א'–ה', 8:00–20:00 בלבד
- תחלופת עובדים בשירות לקוחות: 30–40% בשנה

**מה הלקוחות מתלוננים עליו (על פי סקרי CSAT):**
1. זמן המתנה ארוך
2. להסביר את הבעיה מחדש בכל העברה
3. קבלת תשובות סותרות מנציגים שונים
4. חוסר זמינות מחוץ לשעות הפעילות

### מדוע עכשיו
- LLM בעברית הגיעו לרמת דיוק המאפשרת שיחה שוטפת עם לקוחות
- רגולציית רשות שוק ההון מתגבשת סביב שירות דיגיטלי — "window of opportunity" לפני שיוחמרו הדרישות
- המתחרים (הראל, מגדל, כלל, פניקס) מתחילים ניסויים ראשונים — חלון הזדמנויות של 18–24 חודשים

## 4. מטרות ו-KPIs

### OKR לשנה הראשונה

**Objective:** להפוך לחברת הביטוח עם שירות הלקוחות הזמין, המהיר והעקבי ביותר בישראל.

| Key Result | מדד | יעד |
|------------|-----|-----|
| KR1 | AI Containment Rate (עם Quality Gate) | 50% ב-Phase 1, 65% ב-Phase 2 |
| KR2 | עלות לאינטראקציה | ירידה של 60% לעומת baseline אנושי |
| KR3 | CSAT לאינטראקציות AI | > 4.2/5.0 |
| KR4 | אירועי Compliance | 0 |
| KR5 | זמן פתרון ממוצע | < 4 דקות Tier-1 |

### North Star Metric
**AI Containment Rate with Quality Gate**
= % אינטראקציות שנסגרו ע"י AI בלבד **וגם** קיבלו CSAT ≥ 4/5

*הערה: Containment Rate בלא מדד איכות ניתן ל-"gaming" — לקוח שמתייאש ומנתק אינו לקוח מרוצה.*

### 8 מדדי KPI

| # | מדד | הגדרה | יעד |
|---|-----|--------|-----|
| 1 | **Escalation Rate by Agent** | % sessions שהועלו לנציג אנושי, לפי סוג סוכן | < 30% (Phase 1), < 15% (Phase 3) |
| 2 | **Cost Per Interaction** | עלות תפעולית כוללת / סה"כ אינטראקציות (ב-₪) | 60% ירידה מ-baseline |
| 3 | **First Contact Resolution (FCR)** | % בעיות שנסגרו במגע ראשון ללא חזרה | > 75% תביעות, > 85% כספים |
| 4 | **Average Resolution Time** | מהודעה ראשונה עד אישור פתרון | < 4 דק' Tier-1, < 8 דק' Tier-2 |
| 5 | **Regulatory Compliance Rate** | % אינטראקציות נקיות מהפרה | 100% — Alert אוטומטי על כל הפרה |
| 6 | **Customer Effort Score (CES)** | "כמה קל היה לפתור את הבעיה?" (1–7) | > 5.5 / 7.0 |
| 7 | **Agent Accuracy Rate** | % תשובות עובדתיות מדויקות (מדגם 10% + automated) | > 99% שאילתות עובדתיות |
| 8 | **Retention Trigger Success** *(Phase 2+)* | % לקוחות בסיכון ביטול שחידשו לאחר אינטראקציה עם AI | > 45% |

## 5. פלחי שוק ופרסונות

### פלח א' — Digital Native (גיל 25–40)
**מי:** עצמאים, שכירים צעירים, בעלי פוליסות בודדות (רכב/שכירות)
**ציפיות:** פתרון מיידי, ללא המתנה, ללא צורך בשיחה
**רגישות:** לא יסבלו בוט שמרגיש בוט — רוצים שפה טבעית
**אסטרטגיית AI:** שיחה חופשית, Chat-first, ניתוב מהיר לפתרון self-service

### פלח ב' — מחזיק פוליסות ותיק (גיל 40–60)
**מי:** לקוחות עם 2–4 פוליסות, רכב + דירה + חיים/בריאות, LTV גבוה
**ציפיות:** יחס אישי, זכרון בין אינטראקציות, מישהו שמכיר אותם
**רגישות:** חשד מסוים ל-AI — "רוצה שיבינו אותי, לא רק ישיבו"
**אסטרטגיית AI:** הצגת תיק לקוח מלא בתחילת השיחה, Escalation זמין ברור

### פלח ג' — לקוח בכיר (גיל 60+)
**מי:** לקוחות ותיקים, לרוב בפוליסות מורכבות (חיים, בריאות, רכב)
**ציפיות:** סבלנות, שפה פשוטה, אפשרות לדבר עם אדם
**רגישות:** נמוכה מאוד לאוטומציה — זכויות מוגברות ברגולציה
**אסטרטגיית AI:** Escalation מהיר יותר, Supervisor alert מוגבר, שפה פשוטה

### פלח ד' — לקוח עסקי / SMB
**מי:** עסקים קטנים עם פוליסות מבנה + אחריות מקצועית + רכב צי
**ציפיות:** מהירות, דיוק, ממשק ל-API (עתידי)
**רגישות:** שגיאה בנתוני כיסוי עסקי = חשיפה משפטית
**אסטרטגיית AI:** Escalation מהיר לנציג מומחה, Phase 2+

## 6. הצעת ערך

### ללקוח
| לפני | אחרי |
|------|------|
| המתנה 8–12 דקות | תגובה מיידית, 24/7 |
| מספר את הבעיה מחדש בכל העברה | תיק לקוח טעון — הסוכן יודע מי הוא |
| תשובות סותרות בין נציגים | עקביות מוחלטת, מבוססת נתוני מערכת |
| "צור קשר בשעות הפעילות" | זמין גם בשעות הלילה |

### לחברה
- **חיסכון:** 60% ירידה בעלות לאינטראקציה, ניתוב 70% מהפניות לAI
- **הכנסות:** Retention Agent מוריד Churn; Sales Agent מנצל כל שיחה ל-Cross-sell
- **נתונים:** כל אינטראקציה הופכת למידע מובנה לשיפור חיתום ומוצרים
- **סיכון:** שכבת Supervisor מונעת חשיפה משפטית וחסוך Compliance

### לרגולטור / משפטי
- גילוי AI אחיד בכל שיחה — ללא חריגים
- לוג מלא לכל אינטראקציה, עם timestamp ו-audit trail
- אפס סטייה מהחוק — Supervisor חוסם הפרות לפני שהן מגיעות ללקוח

### 7.2 Orchestrator Agent

**תפקיד:** "שוער" המערכת — מזהה, מאמת, מבין, ומנתב. **לא פותר בעצמו.**

#### זרם לקוח קיים

```
1. גילוי AI חובה (רגולציה)
2. זיהוי: "מה מספר הת.ז. שלך?"
3. אימות: OTP לטלפון רשום / שאלת אימות
4. טעינת תיק לקוח (ראה סעיף 7.5)
5. האם יש קריאה פתוחה במערכת?
   כן → "ראיתי שיש פנייה פתוחה בנושא X, האם זה מה שרצית לברר?"
   לא → "במה אוכל לעזור?"
6. סיווג Intent + ציון ביטחון
7. ניתוב לסוכן מתאים + העברת Context Object
```

#### זרם לקוח חדש

```
1. גילוי AI חובה
2. "אני רוצה לבטח..." → זיהוי ענף:
   - רכב → Sales Agent (vehicle flow)
   - דירה / תוכן → Sales Agent (home flow)
   - מבנה → Sales Agent (building flow) + flag: עשוי להיות מסחרי
   - בריאות / חיים → Sales Agent + disclosure מוגבר
   - פנסיה / השקעות → BLOCK אוטומטי → "אעביר אותך לנציג מורשה"
3. איסוף: שם, טלפון, מייל
4. ניתוב עם context מינימלי
```

#### לוגיקת ניתוב לפי Confidence Score

| Confidence Intent | פעולה |
|-------------------|-------|
| > 85% | ניתוב ישיר לסוכן |
| 60–85% | שאלת הבהרה אחת → ניתוב |
| < 60% אחרי 2 ניסיונות | Escalation לאנושי |
| Intent רגיש (עו"ד, ביטול, תרמית) | Escalation מיידי |

### 7.3 Supervisor Agent — שכבת פיקוח

**תפקיד:** שכבת Guardrail עצמאית שרצה על **כל תשובה** לפני שהיא נשלחת ללקוח. **אינה נראית ללקוח.**

**קלט:** [שאלת לקוח] + [תשובה מוצעת מהסוכן] + [תיק לקוח]
**פלט:** APPROVE / BLOCK + סיבה + ניסוח חלופי (אופציונלי)

#### סט החוקים (Rule Set)

| חוק | תיאור | תגובה |
|-----|--------|--------|
| **No Hallucination** | כל עובדה (סכום, תאריך, שם, כיסוי) חייבת להתבסס על שדה מתיק הלקוח. אין יצירת עובדות חופשית. | BLOCK → ניסוח מחדש בלי העובדה |
| **No Investment/Pension Advice** | כל ניסוח שמרמז על המלצה: "כדאי לך", "עדיף", "מומלץ", תשואות, השוואת קרנות — חסום | BLOCK + Escalation לנציג מורשה |
| **No Coverage Determination** | הסוכן לא מאשר/שולל כיסוי: "התביעה מכוסה" / "זה לא מכוסה" — רק מידע תיאורי | BLOCK → ניסוח עם disclaimer |
| **No PII Exposure** | ת.ז., כרטיס אשראי, IBAN לא מופיעים בתשובה | BLOCK אוטומטי + Security Alert |
| **Mandatory AI Disclosure** | תחילת כל שיחה: "אני מערכת AI של [שם החברה]" | BLOCK אם חסר |
| **Regulatory Compliance** | גילוי נאות (disclosure) נדרש לפני כל הצעת מחיר; חלונות תגובה חוקיים לתביעות מוגדרים כ-hard rules | BLOCK + log לצוות Compliance |
| **Fraud Detection** | Pattern חריג: שינוי פרטי תשלום + פתיחת תביעה באותה שיחה; כתובת משלוח חדשה + בקשת מסמכים רגישים | FLAG → Human Review + Hold |
| **Tone Guard** | תשובה עוינת, פוגענית, מתנשאת, או לא מקצועית | BLOCK + ניסוח מחדש |

#### לוגיקת תגובה

```
APPROVE → התשובה נשלחת ללקוח
BLOCK + יש ניסוח חלופי → הסוכן שולח את הניסוח החלופי
BLOCK + אין ניסוח חלופי (2 ניסיונות) → Orchestrator → Escalation לאנושי
BLOCK + Fraud/Security → Hold מיידי + Human Alert
```

**ממשל:** סט החוקים הוא **Living Document**. בעלים: Compliance Officer + Legal + Head of Products. עדכון: לאחר כל Sprint ובכל שינוי רגולטורי.

### 7.4 מה הסוכנים **לא** עושים — Compliance Boundaries

#### האיסור המוחלט: ייעוץ פנסיוני / ייעוץ השקעות

הסוכן **אינו** יועץ פנסיוני ואינו יועץ השקעות. זו אינה רק מגבלה טכנית — זו דרישה משפטית מפורשת (חוק הסדרת העיסוק בייעוץ השקעות, תשנ"ה-1995; חוק הפנסיה).

| אסור לחלוטין | מותר (מידע עובדתי בלבד) |
|--------------|--------------------------|
| "כדאי לך לעבור לקרן X" | "יש לך קרן פנסיה X שמסתיימת בתאריך Y" |
| "תשואת קרן X גבוהה יותר" | "אם תרצה לדון באפשרויות, אעביר לנציג מורשה" |
| "מומלץ להגדיל את הכיסוי" | "הפוליסה שלך מכסה עד סכום Z" (מתיק הלקוח) |
| השוואת קרנות / מוצרים פיננסיים | — |

כל שאלה המערבת המלצה פיננסית → Supervisor חוסם → Orchestrator → Escalation לנציג בעל רישיון.

### 7.5 תיק לקוח (Customer Profile)

המוצר הנתוני המרכזי — "תעודת הזהות" של הלקוח בשיחה.

**מקורות נתונים:**

| מקור | נתונים |
|------|---------|
| מערכת פוליסות (Synergy / SHAAM) | פוליסות פעילות, תאריכי חידוש, פרמיות, כיסויים |
| מערכת תביעות | תביעות פתוחות וסגורות, סטטוס, סכומים, שמאים |
| Billing / תשלומים | שיטת תשלום, היסטוריה, חובות, אמצעי גבייה |
| CRM | היסטוריית פניות, נציגים שטיפלו, תלונות |
| מערכת תקשורת | שיחות קודמות, מיילים, צ'אטים |

**שדות מרכזיים:**

```
תיק לקוח {
  זהות: { שם, ת.ז., תאריך_לידה, גיל_פנסיה }
  פורטפוליו_פוליסות: [
    { סוג, מספר_פוליסה, סטטוס, פרמיה_חודשית,
      תאריך_חידוש, כיסויים_עיקריים, flag_מסתיים_ב30_יום }
  ]
  תביעות: [
    { מספר, תאריך_פתיחה, סטטוס, סכום, שמאי, מסמכים_חסרים }
  ]
  תשלומים: { שיטה, יום_חיוב, עיכובים, חוב_פתוח }
  LTV_tier: [Bronze | Silver | Gold | VIP]
  flags: [
    at_risk_churn, premium_customer, complaint_on_file,
    minor_customer, fraud_watch, expiring_policy
  ]
  העדפות_תקשורת: { שפה, ערוץ, שעות_מועדפות }
  אינטראקציות_אחרונות: [ { תאריך, ערוץ, נושא, תוצאה } ]
}
```

**אסטרטגיית נתונים:**
- נתוני זהות + פוליסות: Real-time בכל שיחה
- היסטוריית אינטראקציות: Cache עם TTL של 4 שעות
- LTV tier: מחושב לילי
- Integration Middleware: **Phase 0 — חסם לכל ה-MVP**

### 7.6 הסוכנים המומחים

#### Sales Agent (סוכן מכירות)

**מופעל כאשר:** לקוח חדש בוחר ענף ביטוח, או Orchestrator מזהה הזדמנות Cross-sell

**יכולות:**
- הצגת הצעת מחיר Real-time מ-Pricing Engine (API)
- כיסוי מוצר לפי ענף: רכב, דירה, מבנה, בריאות, חיים
- Cross-sell: "ראיתי שיש לך ביטוח רכב אבל אין ביטוח דירה — אוכל להציע?"
- שמירת הצעה ו-Follow-up Async

**גבולות:**
- לא מוכר מוצרי השקעה/פנסיה — Supervisor חוסם
- לא "מחייב" פוליסה — הצגה בלבד; נציג מורשה מאשר וחותם
- לא חורג מתעריפים מוגשים ללא אישור חיתום

**נתונים:** תיק לקוח (פוליסות קיימות, פערי כיסוי), Pricing Engine, Product Catalog

---

#### Retention Agent (סוכן שימור לקוחות)

**מופעל כאשר:** לקוח אומר "רוצה לבטל" / פוליסה מסתיימת תוך 30 יום ואין חידוש / תשלום נכשל פעמיים ב-90 יום

**מילות מפתח לטריגור:** "לבטל", "לא מעוניין", "עוזב", "מחיר גבוה", "מצאתי זול יותר"

**יכולות:**
- הצגת ערך הפוליסה הנוכחית (מה הלקוח מאבד)
- הצעת שינוי מסלול / כיסוי כחלופה לביטול
- הנחת שימור עד X% (סף מוגדר, מאושר מראש)
- Escalation לנציג שימור בכיר ל-LTV גבוה

**גבולות:**
- הנחה מעל הסף — דורשת אישור Supervisor ו/או נציג בכיר
- לא מציע פוליסה חדשה בשיחת שימור (Sales Agent אחר)

---

#### Finance Agent (סוכן כספים)

**מופעל כאשר:** בקשה לשינוי אמצעי תשלום, שאלה על חשבונית, בקשת החזר, בעיית חיוב

**יכולות:**
- צפייה בהיסטוריית תשלומים
- שינוי יום חיוב (OTP נדרש)
- עדכון אמצעי תשלום (OTP נדרש; מספר כרטיס אשראי אינו מוצג בשיחה — PCI-DSS)
- שליחת קבלה/חשבונית במייל
- פתיחת בקשת החזר עד X ₪

**גבולות:**
- החזר מעל X ₪ — דורש אישור נציג אנושי
- Supervisor חוסם כל חשיפת מספרי כרטיס/IBAN בתשובה

---

#### Claims Agent (סוכן תביעות)

**מופעל כאשר:** לקוח מדווח על אירוע, שואל על תביעה קיימת, מבקש הנחיות הגשה

**יכולות:**
- **FNOL (First Notice of Loss):** איסוף מובנה של פרטי האירוע (תאריך, מיקום, תיאור, צד שלישי, מספר משטרה אם רלוונטי)
- עדכון סטטוס תביעה קיימת מה-Claims System
- הנחיות מסמכים: מה חסר, איך להגיש, צ'קליסט
- תיאום פגישה עם שמאי — חלונות זמן מהיומן
- שליחת SMS/מייל אישור פתיחת תביעה (חובה רגולטורית)

**גבולות:**
- **לא** מאשר ולא דוחה תביעה — נדרש שמאי מורשה
- **לא** מאשר כיסוי ("זה מכוסה בפוליסה שלך") — Supervisor חוסם
- **לא** מנהל משא ומתן על סכומי פיצוי

**חובות רגולטוריות (חוק חוזה הביטוח, תשמ"א-1981):**
- אישור קבלת FNOL ללקוח תוך 72 שעות — hard rule מקודד ב-Supervisor

---

### 7.7 פרוטוקול העלאה לנציג אנושי — פרמטרים מלאים

#### קטגוריה 1 — Immediate Escalation (ללא שיקול דעת, אוטומטי)

- לקוח מבקש במפורש: "רוצה לדבר עם נציג", "תעביר אותי לאדם"
- מילות מפתח משפטיות: עורך דין, תביעה משפטית, בית משפט, רגולטור, פיקוח ביטוח
- לקוח קטין (מזוהה לפי ת.ז.)
- אירוע חירום פעיל: תאונה עם נפגעים, שריפה בזמן אמת, פריצה
- כל בקשת ייעוץ פנסיוני / ייעוץ השקעות
- Fraud Flag מה-Supervisor

#### קטגוריה 2 — Confidence-Based Escalation

- ציון ביטחון Intent < 70% לאחר 2 ניסיונות הבהרה
- Supervisor חסם 2 פעמים ואין ניסוח חלופי
- מערכת Backend לא זמינה — תיק לקוח לא נטען

#### קטגוריה 3 — Sentiment/Behavior Escalation

- Sentiment Score < -0.7 (כעס קיצוני, מצוקה, בכי)
- 3+ הודעות עוינות ברצף
- לקוח חוזר על אותה בקשה 3+ פעמים ללא פתרון

#### קטגוריה 4 — Business Rules Escalation

- לקוח VIP (Gold / Platinum LTV) + מצב מורכב
- תביעה מעל X ₪ (ערך מוגדר per product line)
- בקשת ביטול פוליסה עם פרמיה שנתית > Y ₪
- Commercial / SMB — כל מצב מעבר לשאילתה פשוטה

#### מה קורה בהעברה

1. Orchestrator מודיע ללקוח: "אני מעביר אותך לנציג שיוכל לעזור — לא תצטרך לחזור על הבעיה"
2. הנציג האנושי מקבל: תמליל שיחה מלא + תיק לקוח + סיבת ההעברה + urgency_flag
3. SLA: שעות פעילות < 3 דקות המתנה; מחוץ לשעות → Ticket עם SLA חזרה תוך 4 שעות

## 9. רואדמאפ שחרור

### Phase 0 — תשתית (חודש 1–2, לפני MVP)
**מה:** בניית Integration Middleware לתיק לקוח — חיבור למערכות Legacy (SHAAM, Billing, Claims)
**למה זה Phase 0:** ה-MVP לא יכול לרוץ ללא גישה ל-Real-time data. זה ה-blocker הטכני המרכזי.
**Gate:** תיק לקוח נטען בהצלחה לפחות 95% מהזמן עם P95 latency < 1.5 שניות

---

### Phase 1 — MVP (חודשים 3–8)

**עיקרון:** עומק לפני רוחב.

**מה נכנס:**
- Orchestrator Agent (לקוחות קיימים בלבד, עברית)
- Claims Status Agent (שאילתות מצב תביעה + FNOL intake)
- Supervisor Agent (כל 8 חוקים)
- Chat Widget באתר בלבד
- Escalation לאנושי לכל intent אחר

**מה לא נכנס:**
- Sales Agent, Retention Agent, Finance Agent
- זרם לקוח חדש
- ערוץ טלפון
- ערבית / רוסית

**למה Claims כ-MVP:**
- נפח הפניות הגבוה ביותר בשירות לקוחות ביטוח
- Read-only — אין סיכון עסקי/כספי/משפטי
- הצלחה מדידה אובייקטיבית: "קיבל הלקוח את הסטטוס הנכון?"
- מייצר Training Data לשיפור Phase 2

**Gate מעבר ל-Phase 2:**

| תנאי | יעד |
|------|-----|
| אינטראקציות מטופלות | > 500 |
| CSAT | > 4.0 / 5.0 |
| Escalation Rate | < 40% |
| אירועי Compliance | 0 |
| Uptime | > 99.5% |
| Agent Accuracy | > 99% |

---

### Phase 2 — הרחבה (חודשים 9–14)

- Finance Agent (שאילתות + שינויים עם OTP)
- Retention Agent (יירוט ביטולים, חידוש)
- הרחבת Claims → FNOL מלא (לא רק intake)
- זרם לקוח חדש: רכב בלבד
- שילוב ערוץ טלפון / IVR
- שפה: ערבית

---

### Phase 3 — פלטפורמה מלאה (חודשים 15–20)

- Sales Agent (כל ענפים, ללא פנסיה/השקעות)
- פניות יזומות Proactive: פוליסות פגות, תזכורות תשלום
- Agent-to-agent routing (ללא חזרה ל-Orchestrator)
- שפה: רוסית
- Self-improvement loop: הסוכן לומד מ-Escalations ומשפר Confidence

## 11. נספחים

### נספח א — מטריצת ציות רגולטורי
*(יש להשלים עם Compliance Officer לפני Phase 1)*

| יכולת סוכן | בסיס חוקי | דרישת גילוי | דרישת פיקוח אנושי |
|------------|-----------|-------------|-------------------|
| הצגת הצעת מחיר | חוק סוכני הביטוח | גילוי נאות לפני הצעה | נציג מורשה לחתימה |
| FNOL | חוק חוזה הביטוח תשמ"א | אישור קבלה תוך 72 שעות | שמאי לאישור/דחייה |
| שינוי תשלום | — | OTP | לא |
| ייעוץ פנסיוני | חוק הפנסיה + ניירות ערך | — | **חסום לחלוטין** |

### נספח ב — ארכיטקטורת אינטגרציות
*(יש להשלים עם CTO)*

| מערכת | API זמין | Latency יעד | Fallback |
|--------|----------|-------------|----------|
| SHAAM / פוליסות | לבדיקה | < 800ms | Cache 4h |
| Claims System | לבדיקה | < 1s | "אעביר לנציג" |
| Billing | לבדיקה | < 500ms | Cache 4h |
| Pricing Engine | כן | < 2s | "אשלח הצעה במייל" |

### נספח ג — עיצוב שיח (Voice & Tone)
- שפה: עברית רשמית-חמה (לא בירוקרטית, לא חבריית מדי)
- פנייה: "אתה/את" לגיל 18–55; "כבוד" לגיל 60+
- שיחות רגשיות (תאונה, מוות, מחלה): אמפתיה מפורשת לפני פתרון, Escalation מהיר יותר
- מיקס שפות: אם הלקוח כותב Hebrew-English — לענות בעברית ולא לחקות את המיקס

### נספח ד — תכנית בדיקות
1. **Synthetic Testing:** 1,000 שיחות סינתטיות לפני Beta, כולל Red-Teaming (ניסיון לחלץ PII, ייעוץ השקעות, הכרזת כיסוי)
2. **Compliance Audit:** בדיקת כל 8 חוקי Supervisor לפני Launch
3. **Beta מבוקרת:** 5% מהתנועה, לקוחות מוזמנים, 4 שבועות
4. **A/B Production:** AI vs. Human baseline, 30 יום, מובהקות סטטיסטית נדרשת לפני Scale-up
